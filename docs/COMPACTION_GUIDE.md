# Compaction (컴팩션) 쉬운 설명서

## 한 줄 요약

> 대화가 길어져서 AI의 기억 공간이 부족해지면, **오래된 대화를 요약해서 공간을 확보**하는 과정입니다.

---

## 왜 필요한가?

AI에는 한 번에 처리할 수 있는 텍스트 양의 한계가 있습니다.
이걸 **컨텍스트 윈도우**라고 부릅니다. (예: 128K 토큰)

```
대화 시작                                      대화 계속
[■■■■■■■■■■■■■■■■■■■■■■■■■■■■■■■■░░░░░░░░]
[           사용 중 (110K)          ][여유 (18K)]

→ 여유가 부족해지면 새 대화를 처리할 수 없음!
→ 컴팩션으로 공간을 확보해야 함
```

---

## 전체 흐름 (3단계)

```
① 공간이 부족해짐
   "토큰이 임계치를 넘었다!"

② 메모리 플러시 (flush)
   중요한 내용을 파일에 저장 (memory/2026-04-02.md)
   → 나중에 검색으로 찾을 수 있으니 안심하고 지울 수 있음

③ 컴팩션 (compaction)
   오래된 대화를 요약으로 교체
   → 공간 확보 완료!
```

---

## 컴팩션이 하는 일

### Before: 컴팩션 전

```
[시스템 프롬프트]     2K 토큰
[사용자] 안녕?       0.1K 토큰
[AI] 안녕하세요!     0.2K 토큰      ─┐
[사용자] 프로젝트...  3K 토큰        │ 이 부분을
[AI] 네, 분석하면...  5K 토큰        │ LLM이 요약
[사용자] 그러면...    2K 토큰        │ (20K → 2K)
[AI] 결론적으로...    8K 토큰       ─┘
[사용자] DB는?       1K 토큰        ─┐
[AI] PostgreSQL...   3K 토큰        │ 최근 대화는
[사용자] 감사!       0.1K 토큰      │ 그대로 유지
[AI] 도움이 되어...   0.5K 토큰     ─┘
───────────────────────────────
합계: ~25K 토큰
```

### After: 컴팩션 후

```
[시스템 프롬프트]     2K 토큰
[요약] 프로젝트 분석  2K 토큰        ← 오래된 대화의 요약본
  논의함. 결론은...
[사용자] DB는?       1K 토큰        ─┐
[AI] PostgreSQL...   3K 토큰        │ 최근 대화는
[사용자] 감사!       0.1K 토큰      │ 그대로 유지
[AI] 도움이 되어...   0.5K 토큰     ─┘
───────────────────────────────
합계: ~8.6K 토큰  (공간 확보!)
```

## 어떤 메시지를 남기고 어떤 걸 요약해?

### 메시지 개수가 아니라 "토큰 예산"으로 결정합니다

`keepRecentTokens` 설정값만큼의 토큰을 **최신 메시지부터 역순으로** 채워나갑니다.

```
keepRecentTokens = 10,000 이라면:

최신 메시지부터 거꾸로 올라가면서 토큰을 누적:

  메시지 7 (최신): 4K  → 누적  4K  ✅ 유지
  메시지 6:        2K  → 누적  6K  ✅ 유지
  메시지 5:        3K  → 누적  9K  ✅ 유지
  메시지 4:        4K  → 누적 13K  ❌ 예산 초과! 여기서 끊음
  ─── 경계선 (firstKeptEntryId = 메시지 5) ───
  메시지 3:        8K  → 요약 대상
  메시지 2:        5K  → 요약 대상
  메시지 1:        3K  → 요약 대상
```

### 그래서 남는 메시지 수는 상황마다 다릅니다

```
짧은 메시지 위주 대화:
  "안녕" "네" "뭐해?" "코딩" "재밌어?" "네!"
  → 토큰이 적으니 많은 메시지가 예산 안에 들어옴
  → 최근 20~30개 메시지 유지

긴 메시지 위주 대화:
  "이 코드 500줄을 분석해줘..." "분석 결과는..."
  → 토큰이 크니 적은 메시지만 예산 안에 들어옴
  → 최근 2~3개 메시지만 유지
```

## 임계치와 예약 토큰

### 3개의 구간

```
[           전체 컨텍스트 윈도우 (128K)           ]
[■■■ 사용 가능 (104K) ■■■][soft 4K][reserve 20K]
                             ↑          ↑
                        "곧 찰 것 같아"  "여기는 비워둬야 해"
                        (플러시 시작)    (컴팩션 후 확보 목표)
```

| 구간 | 이름 | 기본값 | 역할 |
|------|------|--------|------|
| 사용 가능 | (계산값) | 104K | 실제 대화에 쓸 수 있는 공간 |
| soft | `softThresholdTokens` | 4K | "곧 찰 것 같으니 플러시 시작" |
| reserve | `reserveTokensFloor` | 20K | "컴팩션 후 최소 이만큼은 비워둬" |

### 트리거 계산

```
플러시 시작 기준 = 전체 - reserve - soft
               = 128K  - 20K    - 4K
               = 104K

→ 104K 토큰 사용하면 플러시 + 컴팩션 시작
```

### reserve가 크면? 작으면?

```
reserve = 40K (크게 설정)
→ 128K - 40K - 4K = 84K에서 이미 트리거
→ 더 자주 컴팩션, 하지만 항상 넉넉한 여유

reserve = 5K (작게 설정)
→ 128K - 5K - 4K = 119K까지 버팀
→ 덜 자주 컴팩션, 하지만 빠듯하게 운영
```

## 공간 확보가 충분하지 않으면?

컴팩션 한 번으로 부족하면 **반복합니다:**

```
① 토큰 체크 → 임계치 초과
② 메모리 플러시 (중요 내용 파일 저장)
③ 컴팩션 (오래된 메시지 요약)
④ 검증: 토큰이 줄었나?
   ├─ 충분히 줄었음 → 완료!
   └─ 아직 부족함 → ①로 돌아가서 반복
```

검증 함수 (`estimateTokensAfterCompaction`)가 **실제로 토큰이 줄었는지** 확인합니다.
줄지 않았으면 결과를 거부하고 다시 시도합니다.

---

## 요약은 누가 해?

**LLM (claude-3-5-sonnet 기본)** 이 합니다.

코드로 텍스트를 자르는 게 아니라, AI에게 "이 대화를 요약해줘"라고 시켜서
맥락을 보존하면서 토큰을 줄입니다.

대화가 아주 길면 **단계적 요약** (`summarizeInStages`)을 합니다:

```
[메시지 1~20]  → 요약 A
[메시지 21~40] → 요약 B       → [요약 A + B + C] → 최종 요약
[메시지 41~60] → 요약 C
```

## 관련 설정 모아보기

설정 경로: `agents.defaults.compaction`

```json
{
  "agents": {
    "defaults": {
      "compaction": {
        "keepRecentTokens": 10000,
        "recentTurnsPreserve": 3,
        "reserveTokensFloor": 20000,
        "customInstructions": "코드 블록은 요약하지 말고 유지해줘",
        "memoryFlush": {
          "enabled": true,
          "softThresholdTokens": 4000,
          "forceFlushTranscriptBytes": "2MB"
        }
      }
    }
  }
}
```

| 설정 | 기본값 | 설명 |
|------|--------|------|
| `keepRecentTokens` | - | 최근 메시지 유지 토큰 예산 |
| `recentTurnsPreserve` | - | 최소 유지할 최근 턴 수 (보조) |
| `reserveTokensFloor` | 20,000 | 컴팩션 후 최소 빈 공간 |
| `customInstructions` | - | 요약 시 추가 지시사항 |
| `memoryFlush.enabled` | true | 플러시 활성화 여부 |
| `memoryFlush.softThresholdTokens` | 4,000 | 플러시 시작 토큰 여유분 |
| `memoryFlush.forceFlushTranscriptBytes` | 2MB | 강제 플러시 바이트 |

## 메모리 플러시 vs 컴팩션

| | 메모리 플러시 | 컴팩션 |
|--|-------------|--------|
| **목적** | 중요 내용을 파일에 백업 | 컨텍스트 공간 확보 |
| **순서** | 먼저 | 나중 |
| **결과물** | `memory/2026-04-02.md` 파일 | 요약 메시지 (compactionSummary) |
| **데이터** | 나중에 검색 가능 | 요약만 남고 원본 삭제 |
| **비유** | 중요한 서류를 금고에 넣기 | 책상 위 메모를 정리해서 버리기 |

둘은 **세트로 동작**합니다. 플러시로 백업한 뒤 컴팩션으로 정리하기 때문에,
요약 과정에서 정보가 사라져도 나중에 `memory_search`로 찾을 수 있습니다.
